# 生产稳定性与故障定位

本说明用于第一版公开测试前的运行检查。客户页面不会显示模型名称、供应商名称、Token 或成本信息。

## 本轮已经加入的保护

- 每次前端操作都会生成独立请求编号，服务器响应头中为 `x-request-id`。
- 同一请求编号在 10 分钟内重复提交时，服务器返回 `409`，避免重复执行和重复扣点。
- 普通对话每位用户每分钟最多 12 次。
- 固定报告每位用户每分钟最多 4 次。
- 文件分析每位用户每 5 分钟最多 3 次。
- 独立行情接口每位用户每分钟最多 30 次。
- AI 单次连接超过 90 秒会结束并返回超时提示；行情连接超过 15 秒会结束。
- 固定报告默认同时生成 2 个章节，在不改变报告结构的情况下缩短等待时间。

这些频率保护保存在当前 Node.js 进程内，适合第一版单机部署。以后扩展为多台服务器时，再迁移到 Redis 等共享限流服务。

## 可选环境变量

```txt
REPORT_CONCURRENCY=2
```

建议保持为 `2`。服务器资源充足且AI接口允许更高并发时可以改为 `3`，不要超过 `3`。

## 健康检查

浏览器或监控工具访问：

```txt
https://agent.717828.xyz/api/health
```

正常时返回：

```json
{
  "status": "ok",
  "timestamp": "服务器时间"
}
```

若返回 `degraded`，说明必要的线上环境变量不完整。该接口只显示整体状态，不公开模型或第三方服务名称。

## 查看线上请求日志

```bash
pm2 logs financial-agent-product --lines 100
```

日志中的重要字段：

- `route`：发生问题的接口。
- `requestId`：这一次操作的编号。
- `durationMs`：服务器处理耗时。
- `status`：HTTP 状态码，`200` 为成功，`429` 为操作过快，`502` 通常为外部服务异常。
- `errorMessage`：仅写入服务器日志的真实错误，不展示给普通客户。

出现问题时，可按请求编号筛选：

```bash
pm2 logs financial-agent-product --lines 500 | grep '请求编号'
```

## 更新线上服务器

```bash
cd /var/www/financial-agent-product
git pull origin main
npm install
npm run build
pm2 restart financial-agent-product --update-env
pm2 save
curl -I https://agent.717828.xyz/api/health
```

## 本轮验收项目

1. 快速连续点击两次发送，只出现一次有效回答并只扣一次点数。
2. 普通对话、固定报告、文件分析和行情查询均能完成。
3. AI请求失败时不扣点；重新操作可以正常使用。
4. 访问健康检查接口返回 `200` 和 `status: ok`。
5. PM2日志中可以看到接口名称、请求编号、耗时和状态码。
6. 生成固定报告的等待时间较上一版缩短，章节顺序保持正确。

## 后续升级边界

正式接入付费订单前，需要增加数据库级订单幂等、支付回调验签和自动补偿。当前请求防重复适用于第一版内测和免费点数，不替代支付订单的防重复机制。
